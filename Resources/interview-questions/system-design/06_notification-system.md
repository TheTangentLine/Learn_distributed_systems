# System Design: Notification System

> Classic difficulty: Medium. Tests fan-out, at-least-once delivery, idempotency, and platform-specific push.

---

## Step 1 — Clarify Requirements

**Functional:**
- Send notifications triggered by in-app events (e.g., "Alice liked your photo", "Your order shipped")
- Channels: push (mobile), email, SMS (clarify which are required)
- For this walkthrough: mobile push (iOS + Android) as the primary channel; email as secondary
- User notification preferences: users can opt out per notification type

**Non-functional:**
- Delivery latency: best-effort < 10 seconds for mobile push; up to 1 minute for email
- Scale: 10M notifications/day ≈ 115/sec average; 10,000/sec peak (e.g., breaking news, viral event)
- At-least-once delivery — it is better to deliver a notification twice than not at all
- Idempotency — duplicate deliveries must not cause duplicate billing events or user actions

**Out of scope:** notification templates/localisation, A/B testing, analytics dashboard

---

## Step 2 — Estimate Scale

```
Average rate: 10M/day ÷ 86,400 sec = 115 notifications/sec
Peak rate:    10,000/sec (viral events)

Fan-out:
  A "breaking news" event may target 100M users → need to fan-out quickly
  At 10,000/sec delivery rate: 100M ÷ 10,000 = 10,000 seconds ≈ 2.75 hours
  → Need horizontal scaling of workers + priority queues

Storage per notification:
  { user_id, type, payload, status, idempotency_key, created_at } ≈ 500 bytes
  10M/day × 500 bytes = 5 GB/day → 1.8 TB/year (retain 30 days = 150 GB)

Push provider throughput:
  APNs: ~300K/sec
  FCM: ~500K/sec
  → Each provider can handle our peak load; bottleneck is our own worker fleet
```

## Step 3 — High-Level Design

```
Event Source (order service, social service, etc.)
  │
  ▼
Notification Service (API)
  │  1. Validate & enrich
  │  2. Check user preferences
  │  3. Persist to DB (source of truth)
  │  4. Publish to queue
  │
  ▼
Job Queue (Kafka)
  │
  ├──────────────────┬────────────────
  ▼                  ▼
Push Worker      Email Worker
  │                  │
  ├── iOS →  APNs    └── SMTP / SendGrid
  └── Android → FCM
```

**Key design decision:** The Notification Service writes to the DB before publishing to the queue. If the service crashes after the DB write but before the queue publish, a background reconciliation job re-queues any notifications stuck in `status=PENDING` for > 30 seconds.

## Step 4 — Deep Dive

### Platform-Specific Push

**iOS (Apple Push Notification service — APNs):**
- Requires a device token registered by the iOS app at startup
- HTTP/2 connection to APNs with a JWT credential
- APNs does NOT store and forward — if the device is offline, the notification is lost (APNs will retry for up to 30 days only for "background" notifications)

**Android (Firebase Cloud Messaging — FCM):**
- Similar token-based system; Google stores and forwards for offline devices
- FCM retries delivery for up to 4 weeks

**Device Token Management:**
```
Table: device_tokens
  user_id, device_id, platform (ios/android), token, last_active_at, is_active
```

Tokens expire/change when the app is reinstalled. Handle `InvalidRegistration` or `NotRegistered` errors from APNs/FCM by marking the token as inactive.

### Fan-out Architecture

**Small fan-out (1 user, 1 notification):** direct Kafka message → worker → push provider.

**Large fan-out (1 event → 100M users):**

```
Event: "Breaking News: earthquake M7.0"
  │
  ▼
Fan-out Service reads target user segments in batches of 10,000
  │
  └── For each batch: publish N individual notification messages to Kafka
        Worker fleet (100 workers) consumes messages
        Each worker: look up device token → call push provider API
```

**Priority queues:** use separate Kafka topics by priority:
- `notifications.high` — transactional (order confirmed, password reset) → process first
- `notifications.normal` — social (like, follow, comment)
- `notifications.low` — marketing, bulk

Workers consume `high` first; if no high-priority messages, consume `normal`, then `low`.

### Idempotency

**Problem:** the Kafka consumer crashes after calling the push provider but before committing the Kafka offset. On restart, it re-processes the same message → duplicate notification.

**Solution: deduplication by `(user_id, notification_id)`**

Each notification has a unique `idempotency_key` (generated by the event source or as a UUID at creation).

```go
// Before calling push provider:
exists, _ := db.Get("dedup:" + notification.IdempotencyKey)
if exists {
    log.Info("duplicate notification, skipping", "key", notification.IdempotencyKey)
    return nil  // ACK the Kafka message without delivering
}

// After successful delivery:
db.SetEx("dedup:"+notification.IdempotencyKey, "1", 48*time.Hour)
```

Store dedup keys in Redis with a 48-hour TTL (longer than any reasonable retry window).

**Reference:** Week 7, Day 32 (Idempotency and exactly-once semantics)

### Retry with Exponential Back-off

Push providers return transient errors (503, rate limit). Do not retry immediately — use exponential back-off:

```go
const maxRetries = 5

func deliverWithRetry(n Notification) error {
    for attempt := 0; attempt < maxRetries; attempt++ {
        err := pushProvider.Send(n)
        if err == nil {
            return nil
        }
        if isPermanentError(err) {  // e.g., InvalidDeviceToken
            markTokenInactive(n.DeviceToken)
            return nil  // don't retry permanent errors
        }
        wait := time.Duration(math.Pow(2, float64(attempt))) * time.Second
        time.Sleep(wait)  // 1s, 2s, 4s, 8s, 16s
    }
    return moveToDeadLetterQueue(n)
}
```

**Dead-letter queue (DLQ):** messages that fail after max retries go to a DLQ for manual inspection and alerting.

### User Preference Checks

Before fan-out, filter by user preferences:

```sql
-- Do not send if user opted out of this notification type
SELECT 1 FROM notification_preferences
WHERE user_id = ? AND notification_type = ? AND opted_out = true
```

Cache preferences in Redis (TTL=5min) to avoid a DB hit per notification.

### Store-and-Deliver for Offline Users

For email and in-app notifications: always store in DB and deliver when online.
For mobile push: the push provider (FCM) handles storage for Android; for APNs, notifications can be stored in-app via a catch-up API endpoint (similar to chat system's catch-up query).

## Step 5 — Failure Modes

| Component | Failure | Impact | Mitigation |
|-----------|---------|--------|-----------|
| Push provider down (APNs/FCM) | Notifications cannot be delivered | All pushes fail silently | Retry queue with exponential back-off; DLQ for failed messages; alert on DLQ depth |
| Kafka consumer crashes mid-processing | Message re-delivered | Duplicate notification | Idempotency key + dedup in Redis before calling provider |
| Notification Service crashes before writing to queue | Notification "lost" | User never receives it | Write to DB first (status=PENDING); background reconciliation re-queues PENDING notifications older than 30s |
| Stale device token | Push provider returns `InvalidRegistration` | Silent delivery failure | Mark token inactive on that error code; prompt user to re-register on next app open |
| Fan-out worker fleet overwhelmed | High-priority notifications delayed | Transactional notifications (order confirmed) delayed behind marketing bulk | Priority Kafka topics; high-priority topic read first by all workers |

## Key Talking Points for the Interview

1. **At-least-once + idempotency = effectively-once.** Explain this pattern clearly — the interviewer will probe it.
2. **Priority queues** for different notification types. A marketing blast must not delay a password reset.
3. **Fan-out via a job queue** decouples the event source from delivery — the order service does not need to know about APNs.
4. **Platform differences matter:** APNs does not retry for offline devices the same way FCM does. Showing this knowledge impresses interviewers.
5. **Dead-letter queue** is the production safety net — always mention it when discussing retry logic.
